# 一、先把最终模型压成一句话

新的岗位匹配模型，不再是：

> 岗位 = 一堆平铺元素

而是：

> **岗位 = 一组要求单元（requirement units）**
> 每个要求单元内部再包含：
>
> * 要素集合
> * 逻辑关系
> * 层级关系
> * 强弱程度
> * 单元权重

然后匹配时：

> **先匹配要求单元，再匹配单元内部要素，最后汇总成岗位总分。**

# 三、要求单元的标准逻辑类型

建议固定以下逻辑类型，不要无限发散：

## 1. `SINGLE`

单项要求
例：必须有 C++

## 2. `OR`

任选其一
例：Python or Java

## 3. `AND`

必须同时满足
例：Python + SQL

## 4. `AT_LEAST_K`

至少 K 项
例：Kafka / Redis / MySQL 至少两项

## 5. `PARENT_ANY_CHILD`

父类能力要求
例：熟悉一门主流编程语言

## 6. `PARENT_WITH_CHILD_BONUS`

父类满足即可，但某个子类优先
例：熟悉一门主流编程语言，C++ 优先

这 6 类已经能覆盖你现在提到的大部分真实情况。

# 四、层级命中规则

这是系统最容易做错的地方，必须写死。

## 规则 1：子类可以向上覆盖父类

如果岗位要求：

* 主流编程语言

候选岗位命中：

* Python / Java / C++ / Go

都可视为满足。

---

## 规则 2：父类不能向下覆盖子类

如果岗位要求：

* 必须有 C++ 经验

候选岗位只写：

* 熟悉主流编程语言

不能视为满足 C++。

---

## 规则 3：兄弟节点不能互相替代

如果岗位要求：

* 必须有 C++

候选岗位只有：

* Java

不能满足。

---

## 规则 4：父类命中可作为弱相关信号

如果岗位要求：

* C++

候选岗位只有：

* 主流编程语言

可以给极低弱相关分，或直接 0。
这个要看你的业务保守程度。
如果你追求“必须精确”，这里建议直接记 0。

# 五、召回怎么做

现在进入最重要的工程环节。

你问的核心就是：

> 引入要求单元、OR、层级之后，到底怎么召回？

我建议做 **五路召回**。

---

## 召回路 1：精确 pattern 召回

### 目标

先看是否存在“结构完全一致”或“结构高度相似”的历史岗位 pattern。

### 实现

给每个岗位生成 pattern signature，例如：

```text
must:
  OR(主流编程语言)
  SINGLE(后端开发)
preferred:
  SINGLE(C++)
  SINGLE(推荐系统)
```

查询新岗位时，先去 pattern 索引中查完全相同签名。

### 作用

一旦命中，这通常是最有价值的候选。

---

## 召回路 2：硬要求单元召回

### 目标

用新岗位中最关键的硬要求单元召回候选岗位。

### 例子

新岗位有：

* 必须 C++
* 必须 后端开发
* 必须 分布式系统经验

那就优先用这些单元召回。

### 实现方式

建立：

* 要求单元倒排索引
* 关键 canonical 元素倒排索引

---

## 召回路 3：高价值组合召回

### 目标

解决高频硬要求太泛的问题。

例如：

* “主流编程语言”过于宽泛
* 但“主流编程语言 + 后端开发”
* 或“C++ + 后端开发”
  会非常有区分度

### 实现

构造组合键，例如：

* `TECH_CPP + RESP_BACKEND_DEVELOPMENT`
* `DOMAIN_SECURITY + RESP_BACKEND_DEVELOPMENT`
* `TECH_SQL + DOMAIN_ECOMMERCE`

对这些高价值组合建索引。

---

## 召回路 4：表面表达近邻召回

### 目标

兜底处理：

* 新表达尚未完全归一
* 字典表不完整
* 语义归一暂时失败

### 实现

对岗位全文或关键段落建近邻索引，找到表面上很接近的旧岗位。

### 作用

防漏召回，不让未知表达直接失联。

---

## 召回路 5：层级扩展召回

### 目标

处理“父类要求”和“子类要求”的层级命中。

### 例子

如果岗位要求：

* 主流编程语言

那么召回时可扩展到：

* Python
* Java
* C++
* Go

但如果岗位要求：

* C++

则不应该反向扩展到整个主流编程语言池。

### 原则

层级扩展只能：

* 从父类向下展开子类候选
* 不能从子类向上泛化并替代精确要求

---

# 六、召回键怎么选

不是所有要求单元都同等重要。

建议为每个要求单元计算一个“召回优先分”：

[
召回优先分 = 单元权重 \times 区分度 \times 稳定性
]

其中：

## 单元权重

这个要求本身在岗位里有多重要

## 区分度

这个要求单元在全库中有多稀有、多能缩小候选池

## 稳定性

这个要求单元是否是字典里成熟、边界清晰、不容易误判的单元

---

## 实际选择规则

每个查询岗位只拿：

* 前 3 到 6 个高价值要求单元
* 再加 1 到 3 个高价值组合键
  进入主召回

剩余信息留到精排阶段再用。

---

# 七、打分怎么做

打分分三层。

---

## 第一层：要求单元满足度

这是新的核心。

对查询岗位里的每个要求单元 (u)，计算候选岗位对它的满足度：

[
满足度(u, 候选岗位)
\in [0,1]
]

满足度取决于：

* 逻辑类型
* 层级命中关系
* 子项命中情况
* 是否是精确命中
* 是否只是父类/泛化命中

---

### 1. `SINGLE` 单项要求

例如：

* 必须有 C++

满足度规则：

* 精确命中 C++：1.0
* 只命中父类“主流编程语言”：0 或很低
* 命中兄弟项 Python/Java：0

---

### 2. `OR`

例如：

* Python or Java or C++

满足度规则：

* 命中任意一个：1.0
* 命中多个：1.0 + 小额 bonus
* 一个都没命中：0

注意：
bonus 很小，只用于排序细化，不能改变 OR 的本质。

---

### 3. `AND`

例如：

* Python + SQL

满足度规则：

* 全部命中：1.0
* 命中一半：部分分
* 全缺失：0

例如可设：

[
满足度 = 已命中项数 / 总项数
]

---

### 4. `AT_LEAST_K`

例如：

* Kafka / Redis / MySQL 至少两项

满足度规则：

* 命中数 >= K：1.0
* 命中数 < K：按比例部分给分

例如：

[
满足度 = \min(命中数 / K, 1.0)
]

---

### 5. `PARENT_ANY_CHILD`

例如：

* 熟悉一门主流编程语言

满足度规则：

* 任意一个子类命中：1.0
* 多个子类命中：1.0 + 小额 bonus
* 一个都没命中：0

---

### 6. `PARENT_WITH_CHILD_BONUS`

例如：

* 熟悉一门主流编程语言，C++ 优先

最稳妥的做法：
不要硬塞在一个单元里，直接拆成两个单元：

#### 单元 A

* OR(主流编程语言)
* 硬要求
* 权重高

#### 单元 B

* C++
* 强偏好
* 权重中高

这样最清晰，也最可控。

---

## 第二层：岗位级要求单元总分

有了每个单元的满足度后，就可以做岗位级汇总。

[
岗位要求分 = \sum (单元权重 \times 单元满足度)
]

但这里要分桶：

### 硬要求总分

[
硬要求总分 = \sum (硬要求单元权重 \times 满足度)
]

### 软要求总分

[
软要求总分 = \sum (软要求单元权重 \times 满足度)
]

### 强偏好总分

[
强偏好总分 = \sum (强偏好单元权重 \times 满足度)
]

---

## 第三层：最终岗位匹配总分

[
最终分 =
\alpha \cdot 硬要求总分
+
\beta \cdot 强偏好总分
+
\gamma \cdot 软要求总分
+
\delta \cdot 表面表达接近度
+
\epsilon \cdot 元信息接近度
]

其中一般满足：

* (\alpha) 最大
* (\beta) 次之
* (\gamma) 再次之
* (\delta, \epsilon) 主要用于候选细排

---

# 八、硬要求要不要设“门槛”？

建议要。

因为在真实业务里：

> 某些 must have 没命中，再高的总体相似度也没意义。

所以建议引入“门槛规则”：

## 规则 1

如果关键硬要求单元缺失比例过高，直接降级候选。

## 规则 2

如果某个特别核心的单项硬要求没命中，例如“必须 C++”，则即使其他项很像，也不能排太前。

这能保证系统不会被“看起来像，但核心硬要求不满足”的岗位干扰。

# 九、原来的 ABCDE 怎么迁移到新模型里？

这是你最想搞清楚的地方，我再压一遍。

原来：

* A/B/C/D/E 是直接比

现在：

* A/B/C/D/E 不再直接打总分
* 它们要先被组织进要求单元里

例如原来你会说：

新岗位 = A B C D E
候选岗位 = B C D E
所以候选命中 4/5，很高

但现在可能真实结构是：

## 查询岗位

### 单元 1

* A
* 硬要求
* 权重 0.5

### 单元 2

* OR(B,C,D,E)
* 硬要求
* 权重 0.3

### 单元 3

* F
* 软要求
* 权重 0.2

如果候选岗位只有 B C D E：

* 单元 1 = 0
* 单元 2 = 1
* 单元 3 = 0

最终总分不高，因为最重要的 A 没满足。

所以：

> **元素数量命中，不再直接代表高匹配。**
> **必须先看这些元素属于哪个要求单元，以及这个单元有多重要。**

# 十、权重怎么评估得更细？

现在权重有两层：

---

## 层 1：要求单元权重

表示：

* 这个要求单元在岗位中有多重要

来源：

* must / preferred / nice to have
* 所在 section
* 表达强度
* 是否出现在标题/summary/basic qualifications
* 是否是核心职责
* 是否高频但关键

---

## 层 2：要求单元内部成员权重

表示：

* 在这个单元里，哪个成员更关键

例如：

* OR(Python, Java, C++)
  如果写的是：
* “Python or Java required, C++ is a plus”

那可以拆成：

* OR(Python, Java) 作为硬要求单元
* C++ 作为额外偏好单元

通常不建议在一个 OR 单元里塞太多不对称权重，
最好拆成多个单元，更稳定。

# 十一、完整流程伪代码

下面给你一版简化但够清楚的伪代码。

```python
def match_job(query_job, pattern_index, unit_index, combo_index, surface_index):
    # 1. 抽取表面表达
    surface_elements = extract_surface_elements(query_job)

    # 2. 语义归一
    canonical_elements = canonicalize(surface_elements)

    # 3. 解析要求单元
    requirement_units = build_requirement_units(canonical_elements, query_job)

    # 4. 计算每个要求单元的权重
    weighted_units = assign_unit_weights(requirement_units, query_job)

    # 5. 选择召回键
    recall_keys = select_recall_keys(weighted_units)

    # 6. 多路召回
    candidates = set()
    candidates |= exact_pattern_recall(weighted_units, pattern_index)
    candidates |= unit_recall(recall_keys, unit_index)
    candidates |= combo_recall(recall_keys, combo_index)
    candidates |= surface_recall(query_job, surface_index)
    candidates |= hierarchical_expansion_recall(weighted_units, unit_index)

    # 7. 对候选pattern打分
    scored_patterns = []
    for pattern in candidates:
        score = score_pattern(weighted_units, pattern)
        scored_patterns.append((pattern, score))

    top_patterns = top_k(scored_patterns, k=20)

    # 8. 在pattern内部做岗位实例精排
    scored_jobs = []
    for pattern, pattern_score in top_patterns:
        for job in pattern.member_jobs:
            final_score = score_job_instance(query_job, weighted_units, job, pattern_score)
            scored_jobs.append((job, final_score))

    # 9. 排序输出
    return sorted(scored_jobs, key=lambda x: x[1], reverse=True)
```

# 十二、我对你下一步最建议的落地方向

现在你最该做的不是继续抽象，而是把这三件事先固定下来：

## 1. 要求单元的数据结构

这是整个模型的地基。

## 2. 六类逻辑类型

先不要无限扩展，先固定：

* SINGLE
* OR
* AND
* AT_LEAST_K
* PARENT_ANY_CHILD
* PARENT_WITH_CHILD_BONUS

## 3. 满足度函数

把每一类逻辑的满足度算式写死，跑起来
