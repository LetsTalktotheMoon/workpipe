match_pipe 多层级匹配系统评估

当前系统不是单一的“相似 JD 检索器”，而是一个分层语义匹配器 + 双通道 starter 选择器 + 下游 writer/reviewer/planner 验证链。 语义 teacher 已冻结为 teacher_b_semantic_v1，重点测试正在从“找得准”转向“如何把匹配结果更稳定地用进简历生成”。

teacher frozen = true student ready = true 双通道下游收益尚未稳定单边领先 支持增量入库

3. 匹配规则分层

第 0 层：文本切块 标题、summary、core_skills、must_have_quals、preferred_quals、core_responsibilities、metadata 被拆成 block，不同 block 自带不同约束权重。
第 1 层：canonical alias 识别 文本先映射到 taxonomy canonical id，例如 TECH_PYTHON、TECH_AWS、RESP_SOFTWARE_DEVELOPMENT。
第 2 层：逻辑结构识别 系统显式识别 SINGLE、OR、AND、AT_LEAST_K、PARENT_ANY_CHILD。
第 3 层：unit 权重 must_have > strong_preference > preferred > background，并叠加内容类型、section、logic bonus。
第 4 层：召回 候选来源是 pattern_exact、hard_unit、hierarchy_expand、combo，必要时还有 surface/fallback。
第 5 层：语义打分 总分核心是 requirement_score 和 hard_requirement_score；硬缺口低于阈值会触发惩罚。
第 6 层：双通道选 starter semantic channel 负责“最像、最可复用”；company channel 负责“同公司连续性”。

5. “一份新简历/新职位打进来”时的实际匹配路径

新职位进入
  -> loader 读取 row / raw_text
  -> units.build_structured_job:
       - block 拆分
       - alias 命中 canonical ids
       - 识别 OR / AND / AT_LEAST_K / PARENT_ANY_CHILD
       - 生成 requirement_units / pattern_signature / recall_keys
  -> MatchEngine._candidate_pool:
       - pattern_exact
       - hard_unit
       - hierarchy_expand
       - combo
       - (teacher_a/c 可选 surface / same_company)
  -> MatchEngine._score_candidate:
       - 每个 requirement unit 打分
       - 聚合 requirement_score / hard_requirement_score
       - 若 must gap 大，整体降权
  -> StarterSelector.select:
       - semantic_best_anchor
       - semantic_positive_cluster（若 generic-title 且多阳性可接受）
       - company_best_anchor
       - writer_input.primary_anchor / continuity_anchor
  -> downstream writer / planner:
       - no_starter / old_match / new_dual_channel 对比实验
       - reviewer 打分
       - planner 决定 direct_review / write / reject_starter

如果你说的“新简历打进来”是“新生成的简历成为后续可复用 anchor”，那它不是直接进 semantic teacher 规则层，而是先以 portfolio artifact 的形式被 loader 重新纳入历史池，之后才会在后续 job 匹配中作为候选 starter 被复用。

6. 当前系统的优点

优点 1：语义 teacher 很克制 冻结版 teacher_b_pure_semantic 明确关掉 same-company、surface、metadata、duplicate shortcut，避免检索器被“公司名/标题重合”污染。
优点 2：显式建模逻辑结构 它不是只比关键词重合，还区分 AND/OR/至少 K 个命中，这一点对岗位要求很关键。
优点 3：把公司连续性拆成辅助通道 同公司历史价值被保留，但不混进 semantic teacher 核心，这是架构上正确的边界。
优点 4：支持增量增长 新 job 可即时写入 index，低信号项会 quarantine，未知 surface token 会进入 alias candidate 队列。

7. 当前风险与限制

风险 1：downstream 收益不稳定 小样本里 new_dual_channel_anchor 平均分 94.45，略低于 old_match_anchor 的 94.7，而且 token 成本更高。
风险 2：semantic best 可能不可写 有案例里 semantic best 本身没有 resume artifact，于是 writer 最终退回到 semantic top-k 里第一个可写 anchor，说明“最佳匹配”和“最佳 starter”仍是两件事。
风险 3：generic title 仍难 boundary pool 目前只有 1 个 case，说明“多阳性可接受”机制已经存在，但边界样本仍偏少。
风险 4：planner 还未稳定优于直接 writer planner dual-channel 在个别 case 上提升明显，但也有 case 低于历史 no-starter/old-match。

8. 两个真实样例

样例 系统行为 结论
Color / New Grad Software Engineer 旧路由命中 AWS Generalist Embedded；新双通道的 semantic best 是 Suffolk Construction / AI Engineer， 没有 company anchor，最终按单 anchor 写。 旧流 94.9 分，新双通道 94.0 分。这个例子说明 semantic best 不一定比成熟 seed 更适合作为写作起点。

9. 我对当前系统的判断

架构判断

这套设计方向是对的。语义 teacher 保持纯净，company continuity 被下沉到 reranker / writer 约束层，避免检索器过拟合历史公司与标题重复。 这比“把 same_company 直接混进主分”更稳。

落地判断

真正还没完全收敛的是“如何把 semantic 结果变成最优 starter”。当前系统已经意识到 semantic-best、semantic-starter、semantic-writer-anchor 是三个不同角色， 但这层还在试验，尤其是在 generic title 和 artifact 缺失场景下。

10. 下一步最值得补强的点

补强 1：把 “best match” 和 “best writable anchor” 分开打分 现在已经部分这么做了，但还可以更显式：在 selector 输出里直接区分 semantic_judge_best / semantic_resume_best / writer_best。
补强 2：扩边界池 generic-title、same-company-structure-different、must-vs-preferred-confusing 这几类都应该补更多 case。
补强 3：把 planner 的收益条件化 不要默认 planner 一定介入，而应该在 semantic score、hard gap、artifact quality 达到阈值时才走 planner-heavy 路线。
补强 4：把新历史简历的入池质量门槛显式化 不是所有新生成简历都应直接成为后续 anchor，最好加一个 review_final_score / pass gate。
